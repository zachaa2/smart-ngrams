Algorithm

How smart n-grams are computed from a body of text.

Overview

A traditional n-gram extractor produces every contiguous window of n words from a document. The result is dominated by uninteresting phrases stitched together by high-frequency function words such as the , of , or and . The goal of this project is to extract “interesting” n-grams : contiguous word sequences that do not cross a stop-word boundary .

The pipeline is built from two stages:

  1. Tokenization — the raw text is normalized and chopped into a list of segments . A segment is a maximal run of valid words containing no stop word. Stop words therefore act as delimiters rather than as members of any n-gram.
  2. Counting — unigrams (words) and n-grams of arbitrary order n are tallied per segment. By construction, no n-gram can span a segment boundary, so no n-gram can contain a stop word.
Stop-word list. The implementation uses the 50 most frequent words from the AP89 newswire corpus ( the , of , to , a , and , in , said , for , that , was , ...). The set is hard-coded and treated as a constant.

The remainder of this page presents the algorithm in five self-contained pieces. Each is given as semi-formal pseudocode broken down by the major routines. The actual Go implementation lives in internal/ngrams/ .

Notation

  • T — the input text, a sequence of Unicode characters.
  • S — the set of stop words.
  • A segment is a list of words [w 1 , w 2 , ..., w k ] with w i ∉ S and |w i | > 1 .
  • segments is a list of such segments produced by tokenization.
  • A word is a maximal run of alphabetic characters, optionally containing single quotes (to preserve contractions such as don’t ).

1. Driver

The entry point combines tokenization with one pass per chosen n-gram order. The caller supplies pre-allocated count maps; the algorithms mutate them in place.

Algorithm 1: SmartNgrams (T, N) 
// T: input text. N: set of n-gram orders to compute (e.g. {1,2,3}).
Input:  text T, orders N ⊆ ℕ
Output: a map  n → (map word → frequency)  for each n ∈ N

stopCounts ← empty map
segments   ← Tokenize(T, stopCounts)

for each n in N do
    counts[n] ← empty map
    if n = 1 then
        CountWords(segments, stopCounts, counts[1])
    else
        CountNGrams(segments, counts[n], n)

return counts

Note the asymmetry between n = 1 and n ≥ 2 : unigrams include stop words (because individual words by themselves are still meaningful), whereas n-grams of order two or higher deliberately exclude them.

2. Tokenization

Tokenization is performed in two passes. The first pass ( SegmentText ) scans the raw character stream, builds words, and uses stop words as segment delimiters. The second pass ( HandleQuotes ) cleans up apostrophes that survived the first pass.

Algorithm 2: Tokenize (T, stopCounts) 
segments ← SegmentText(T, stopCounts)
tokens   ← HandleQuotes(segments)
return tokens

2.1 Character-level segmentation

A single left-to-right scan over T assembles words one character at a time. Any character that is neither a letter nor a single quote terminates the current word. Each finished word is then classified:

  • If it is a stop word ⇒ tally it, close the current segment.
  • If it is a single letter ⇒ discard it, close the current segment.
  • Otherwise ⇒ append it to the current segment.
Algorithm 3: SegmentText (T, stopCounts) 
segments       ← [ ]
currentSegment ← [ ]
word           ← ""

procedure HandleWord():
    word ← Trim(word, "'")                  // strip leading/trailing apostrophes
    if word ∈ S then
        stopCounts[word] ← stopCounts[word] + 1
        FlushSegment()
    else if |word| = 1 then
        FlushSegment()                          // single letters also delimit
    else if |word| > 0 then
        append word to currentSegment
    word ← ""

procedure FlushSegment():
    if currentSegment is non-empty then
        append (copy of currentSegment) to segments
        currentSegment ← [ ]

for each character c in T do
    if IsLetter(c) or c = '\'' then
        word ← word + ToLower(c)
    else if |word| > 0 then
        HandleWord()

if |word| > 0 then HandleWord()         // flush trailing word
FlushSegment()                                 // flush trailing segment

return segments

2.2 Handling internal apostrophes

A single apostrophe inside a word is a legitimate contraction ( don’t , it’s ) and is preserved. Two or more apostrophes inside the same token, however, almost always indicate quoted text or punctuation noise; they are treated as a hard delimiter. The post-pass below splits such tokens and closes the surrounding segment when the resulting fragment is only a single character.

Algorithm 4: HandleQuotes (segments) 
updated ← [ ]

for each segment in segments do
    cur ← [ ]
    for each word in segment do
        buf      ← ""
        quoteCtr ← 0
        for each character c in word do
            if c = '\'' then
                quoteCtr ← quoteCtr + 1
                if quoteCtr > 1 then           // second quote => split
                    Emit(buf, cur, updated)
                    buf ← ""; quoteCtr ← 0
                    continue
            buf ← buf + c
        Emit(buf, cur, updated)               // flush remainder of word
    if cur is non-empty then append cur to updated

return updated

procedure Emit(buf, cur, updated):
    if |buf| > 1 then
        append buf to cur
    else if |buf| = 1 then           // single-char fragment delimits
        if cur non-empty then
            append cur to updated; cur ← [ ]
Why two passes? The first pass is character- oriented and only knows about word boundaries imposed by non-letter characters. Apostrophes need word-level context (“how many of them are in this token?”) to be classified correctly, which is exactly what the second pass provides.

3. Counting

3.1 Unigrams

Unigram counts include every multi-character token, plus the stop-word tallies that were collected during segmentation. Single letters are excluded entirely (they have already been dropped by tokenization).

Algorithm 5: CountWords (segments, stopCounts, wordCounts) 
for each (w, c) in stopCounts do
    if |w| > 1 then
        wordCounts[w] ← wordCounts[w] + c

for each segment in segments do
    for each word in segment do
        wordCounts[word] ← wordCounts[word] + 1

3.2 N-grams (n ≥ 2)

For each segment with at least n words, slide a window of width n across it, joining the words with single spaces to form a key. Segments shorter than n are skipped, so this naturally handles e.g. computing 5-grams over a document whose segments are mostly shorter than five words.

Algorithm 6: CountNGrams (segments, nGramCounts, n) 
for each segment in segments do
    if |segment| < n then continue
    for i ← 0 to |segment| - n do
        gram ← segment[i]
        for j ← 1 to n - 1 do
            gram ← gram + " " + segment[i + j]
        nGramCounts[gram] ← nGramCounts[gram] + 1

4. Worked example

Consider the input

"The quick brown fox jumps over the lazy dog, and don't stop."

After SegmentText the stop words the , over ... wait — over is not in the AP89 list, but the and and are. Lower-casing and splitting on the stop words { the , and } produces three segments:

[ ["quick","brown","fox","jumps","over"], ["lazy","dog"], ["don't","stop"] ]

HandleQuotes leaves the result unchanged because don’t has only a single apostrophe. CountNGrams with n = 2 yields:

{ "quick brown":1, "brown fox":1, "fox jumps":1, "jumps over":1, "lazy dog":1, "don't stop":1 }

Notice that “over the” and “the lazy” — the bigrams a naive extractor would produce — are absent: they would have to bridge a stop-word delimiter.

5. Complexity

Let |T| be the number of characters in the input, W the total number of words produced, and n max the largest requested n-gram order.

  • Tokenization performs one linear scan of the character stream plus a linear scan of the words, giving O(|T|) time and O(W) space.
  • Stop-word lookup is O(1) per word using a hash set.
  • Counting for a fixed order n visits each window once and builds a key of length O(n) , giving O(W · n) time. Computing every order up to n max is therefore O(W · n max 2 ) .

In practice n max is small (typically ≤ 5), so the pipeline is effectively linear in the input size.
